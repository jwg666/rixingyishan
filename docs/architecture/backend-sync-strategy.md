# 后端与同步策略（MVP）

## 决策
- **阶段 1（MVP）**：`local-first`，无强制登录；支持本地完整使用
- **阶段 2（增强）**：引入账号与云同步，打通多端数据互通

该策略兼顾“快速上线”与“后续可扩展”，避免一开始被账号体系拖慢节奏。

## 为什么不是一开始就强制登录
- 记录类产品的首要门槛是“马上能记”
- 强登录会增加首日流失
- 本地草稿是弱网与权限受限场景的安全兜底

## 阶段 2 最小后端设计
### 组件
- `API 服务`：记录元数据 CRUD、上传凭证签发
- `对象存储`：图片/视频文件
- `数据库`：记录主表 + 媒体子表 + 上传任务日志

### 最小 API（示意）
- `POST /auth/login`（可选，手机号/微信）
- `POST /upload/policy` 获取临时上传凭证
- `POST /records` 创建记录（包含 media remoteUrl）
- `GET /records?dayKey=yyyy-mm-dd&page=1`
- `GET /records/:id`
- `PATCH /records/:id`

## 同步策略
- 客户端记录保留 `updatedAt` 与 `syncVersion`
- 上行：本地改动优先上传，服务端返回最新版本号
- 下行：按时间窗口拉增量
- 冲突：MVP 采用“最后写入覆盖（LWW）”，并保留冲突日志

## 安全与合规基线
- 上传凭证短时效 + 最小权限
- 服务端校验 MIME、大小、扩展名
- 传输全程 HTTPS
- 预留内容审核接入点（图片/视频可异步审核）

## 演进路径
- MVP：仅本地
- v1：账号 + 云同步 + 跨端拉取
- v1.1：内容审核 + 举报
- v1.2：搜索/标签/统计分析
