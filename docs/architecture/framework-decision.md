# 跨端框架决策：Good Deeds App

## 结论
- 主框架：`uni-app` + `Vue3` + `TypeScript`
- 目标端：`iOS` / `Android` / `H5` / 小程序（优先微信，次优抖音）
- 原则：同一套页面与业务逻辑优先复用，平台差异下沉到能力适配层

## 选择依据
- 团队技术偏好为 `Vue`，迁移成本低，上手速度快
- 在 App + H5 + 小程序同时覆盖时，`uni-app` 的工程链路成熟度和落地经验相对稳定
- 媒体采集、文件读写、上传等能力有跨端 API 与生态插件可用

## 与备选方案对比
### Taro（React 生态）
- 优点：小程序/H5 工程化成熟，React 生态强
- 不选原因：当前团队非 React 主栈，切换成本高，不符合“快启动”目标

### Flutter / React Native + 单独小程序
- 优点：纯 App 体验与性能可做到更高上限
- 不选原因：与“App 与小程序尽量同代码”目标冲突，通常需要两套前端

## 边界与技术红线
- 不在 MVP 内实现复杂视频编辑（滤镜、转场、多轨剪辑）
- 不在 MVP 内追求重动画或复杂长列表的极限性能
- 不在 MVP 内实现多小程序平台深度定制，优先打通微信

## 触发升级条件
出现以下任一情况，需评估原生插件或局部原生化：
- 视频处理成为核心卖点（压缩、转码、编辑链路复杂）
- 复杂页面在低端机持续掉帧，且优化后仍无法达标
- 业务必须依赖某端独占能力，跨端封装成本显著高于收益

## 工程落地约束
- 业务代码不得直接访问平台 API，必须经过 `platform adapter`
- 记录创建链路必须支持离线草稿
- 媒体上传使用统一任务队列，失败可重试
